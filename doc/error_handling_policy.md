# 예외 처리 정책 정의

## 1. 목적

이 문서는 1차 MVP에서 오류를 일관되게 분류하고 처리하기 위한 기준을 정의한다.  
오류 분류는 재시도, 계속 진행, 즉시 실패, 운영 로그 기록 기준으로 사용한다.

## 2. 적용 범위

- 포함: 재시도 가능 오류, 비치명적 오류, 치명적 오류, 설정 오류 분류 기준
- 제외: 실제 외부 API 호출 구현, 엑셀 출력 구현

## 3. 오류 유형

| 오류 유형 | 코드 값 | 의미 |
| --- | --- | --- |
| 재시도 가능 오류 | `retryable` | 재시도 후 복구될 수 있는 일시 오류 |
| 비치명적 오류 | `non_fatal` | 일부 항목 실패 후 전체 실행은 계속 가능한 오류 |
| 치명적 오류 | `fatal` | 실행을 실패로 종료해야 하는 오류 |
| 설정 오류 | `configuration` | 실행 시작 전 설정 검증 실패 |

## 4. 재시도 가능 오류

아래 오류는 재시도 가능 오류로 본다.

- 외부 API 일시 timeout
- 외부 API 429 또는 일시 rate limit
- 외부 API 5xx 계열 일시 장애
- 네트워크 일시 단절
- 공지된 점검 시간 직전/직후의 일시 실패

처리 기준:

- 설정된 `retry_count`만큼 재시도한다.
- 재시도 후 복구되면 성공 흐름으로 계속 진행한다.
- 재시도 후에도 실패하면 해당 상황에 따라 비치명적 또는 치명적 오류로 승격한다.

## 5. 비치명적 오류

아래 오류는 비치명적 오류로 본다.

- 일부 공고의 선택 필드 누락
- 일부 공고의 날짜 파싱 실패
- 일부 공고의 부적격 제외
- 일부 공고의 원천 상세 페이지 접근 실패
- 재시도 후 복구된 오류

처리 기준:

- 해당 항목을 제외하거나 기본값 없이 빈 값으로 유지한다.
- 전체 실행은 계속한다.
- `excluded_count` 또는 오류 metadata에 기록한다.
- 실행 결과는 필요 시 `partial_success`로 기록한다.

## 6. 치명적 오류

아래 오류는 치명적 오류로 본다.

- 기업마당 또는 나라장터 중 활성화된 한 트랙 전체 수집 실패
- DB 저장 실패
- 엑셀 산출물 미생성
- 운영 로그 미생성
- 필수 필드 파싱 실패율이 허용 기준을 초과
- 중복 저장 방지 구조 실패

처리 기준:

- 실행을 실패로 종료한다.
- `run_finished` 이벤트의 `status`는 `failed`로 기록한다.
- `error` 이벤트에 `error_type=fatal`을 기록한다.

## 7. 설정 오류

설정 오류는 실행 시작 전 검증에서 발견되는 오류이다.

예시:

- 활성화된 소스의 endpoint 누락
- 키워드 목록 누락
- 지원하지 않는 `runtime.action`
- 지원하지 않는 저장소 타입
- 유효하지 않은 timezone

처리 기준:

- 외부 API 호출, DB 저장, 엑셀 출력 전에 즉시 실패한다.
- `error_type=configuration`으로 기록한다.
- CLI 종료 코드는 `2`를 사용한다.

## 8. 오류 승격 기준

| 시작 오류 | 승격 조건 | 승격 결과 |
| --- | --- | --- |
| `retryable` | 재시도 횟수 초과 후 소스 전체 실패 | `fatal` |
| `retryable` | 재시도 후 일부 항목만 실패 | `non_fatal` |
| `non_fatal` | 필수 필드 파싱 실패율 기준 초과 | `fatal` |
| `non_fatal` | 엑셀 산출물 생성 불가 | `fatal` |

## 9. 예외 타입 기준

코드에서는 공통 기반 예외 `Project1Error`를 두고 오류 유형을 함께 가진다.

| 예외 타입 | 오류 유형 |
| --- | --- |
| `RetryableError` | `retryable` |
| `NonFatalError` | `non_fatal` |
| `FatalError` | `fatal` |
| `StorageWriteError` | `fatal` |
| `ConfigurationAppError` | `configuration` |

외부 라이브러리 예외는 경계 계층에서 위 예외 유형 중 하나로 변환한다.

## 10. 저장 실패 처리 기준

DB 저장 실패는 치명적 오류로 고정한다.  
저장 실패는 일부 공고 제외나 `partial_success`로 처리하지 않는다.

처리 기준:

- 저장소 계층은 SQLite write 실패를 `StorageWriteError`로 변환한다.
- Collect Use Case는 저장 실패가 발생하면 현재 소스 처리를 즉시 중단한다.
- `RunSummary.status`는 `failed`로 기록한다.
- `SourceRunSummary.error_count`는 저장 실패 1건을 반영한다.
- `error` 이벤트에는 `error_type=fatal`을 기록한다.
- `run_finished` 이벤트의 `status`는 `failed`로 기록한다.

## 11. 1차 확정 사항

- 오류 유형은 `retryable`, `non_fatal`, `fatal`, `configuration`으로 구분한다.
- 설정 오류는 실행 시작 전에 실패한다.
- 활성화된 소스 전체 실패, DB 저장 실패, 엑셀 미생성, 로그 미생성은 치명적 오류이다.
- 일부 선택 필드 누락과 재시도 후 복구된 오류는 비치명적 오류이다.
- 재시도 가능 오류는 재시도 실패 후 상황에 따라 비치명적 또는 치명적 오류로 승격한다.
- DB 저장 실패는 `failed` 상태로 종료하며 `partial_success`로 처리하지 않는다.

## 12. 남은 결정 사항

- 필수 필드 파싱 실패율 계산 방식
- 외부 API별 상세 retry 대상 status code
- 비치명적 오류 누적 시 `partial_success`로 전환하는 세부 기준
